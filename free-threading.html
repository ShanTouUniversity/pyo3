PyO3 user guide

支持自由线程的 CPython

CPython 3.14 声明支持“自由线程”（free-threaded）版本的 CPython，该版本不再依赖全局解释器锁（通常称为 GIL）来确保线程安全。自 0.23 版本起，PyO3 支持为自由线程版本的 Python 构建 Rust 扩展，并支持从 Rust 调用自由线程版本的 Python。

如果你想了解更多关于自由线程 Python 的背景知识，可以参考 3.13 版本发布说明中的新特性部分（这是“自由线程”构建首次作为实验性功能引入），CPython 文档中的自由线程 HOWTO 指南，社区维护的 Python 自由线程指南中的扩展移植指南，以及 PEP 703，该文档提供了 CPython 中自由线程实现的技术背景。

在支持 GIL 的构建中（在“自由线程”构建推出之前是唯一选择），全局解释器锁对 Python 运行时的访问进行了串行化。因此，由于Amdahl 定律，GIL 成为多线程 Python 工作负载并行扩展的根本限制，因为在单一执行上下文中执行并行处理任务的任何时间，从根本上无法通过并行化加速。

自由线程构建消除了这一对多线程 Python 扩展的限制。这意味着使用 Python 的 threading 模块实现并行化变得更加直接。如果你曾经需要使用 multiprocessing 来为某些 Python 代码实现并行加速，那么自由线程很可能允许你改用 Python 线程来完成相同的工作流。

PyO3 对自由线程 Python 的支持将允许编写天生线程安全的原生 Python 扩展，相比 C 扩展能提供更强的安全性保证。我们的目标是通过基于 Rust 的 Send 和 Sync trait，在原生 Python 运行时中实现“无畏并发”（fearless concurrency）。

本文档提供将使用 PyO3 的 Rust 代码移植到自由线程 Python 下运行的建议。

使用 PyO3 支持自由线程 Python

许多简单的 PyO3 使用场景（例如为无副作用的纯 Rust 函数暴露绑定，或定义不可变的 Python 类）在自由线程构建下可能“开箱即用”。你唯一需要做的就是标注项目中由 Rust 代码声明的 Python 模块，表明它们支持自由线程 Python，例如通过 #[pymodule(gil_used = false)] 声明模块。

更复杂的 #[pyclass] 类型可能需要直接处理线程安全；指南中有一个专门章节讨论此问题。

在底层，标注模块会将扩展定义的模块的 Py_MOD_GIL 槽位设置为 Py_MOD_GIL_NOT_USED，这样解释器在运行时就能知道扩展的作者认为该扩展是线程安全的。你应该仅在你确定你的扩展是线程安全的情况下才这样做。由于 Rust 本身的保证，许多扩展已经满足这一点，但请参见下文以了解如何评估现有 Rust 扩展的线程安全性，以及如何理解在没有 GIL 的 Python 运行时中使用 PyO3 API。

如果你未显式标记模块为线程安全，Python 解释器在导入你的模块时将在运行时重新启用 GIL，并打印一条 RuntimeWarning 警告，消息中包含导致 GIL 被重新启用的模块名称。你可以通过将环境变量设置为 PYTHON_GIL=0 或在启动 Python 时传入 -Xgil=0 来强制保持 GIL 禁用（0 表示 GIL 被关闭）。

如果你确定在 PyModule 中暴露的所有数据结构都是线程安全的，则可以在声明模块的 pymodule 过程宏中传入 gil_used = false 参数，或在 PyModule 实例上调用 PyModule::gil_used。例如：

use pyo3::prelude::*;

/// 本模块支持自由线程 Python
#[pymodule(gil_used = false)]
fn my_extension(m: &Bound<'_, PyModule>) -> PyResult<()> {
    // 添加你知道是线程安全的成员到模块
    Ok(())
}

或对于未使用 pymodule 宏构建的模块：

use pyo3::prelude::*;

#[allow(dead_code)]
fn register_child_module(parent_module: &Bound<'_, PyModule>) -> PyResult<()> {
    let child_module = PyModule::new(parent_module.py(), "child_module")?;
    child_module.gil_used(false)?;
    parent_module.add_submodule(&child_module)
}

目前你必须通过标注扩展中定义的模块来显式选择支持自由线程。在未来的 PyO3 版本中，我们计划将 gil_used = false 设为默认值。

参见 string-sum 示例，了解如何使用原始 FFI 调用为使用单阶段初始化的模块声明自由线程支持，以及 sequential 示例了解使用多阶段初始化的模块。

如果你想通过条件编译在自由线程构建下触发不同的代码路径，可以使用 Py_GIL_DISABLED 属性，前提是你的 crate 已配置生成必要的构建配置数据。详见指南章节，了解支持多个不同 Python 版本（包括自由线程构建）的更多细节。

自由线程构建的特殊注意事项

自由线程解释器没有 GIL。许多现有扩展提供的可变数据结构依赖 GIL 来锁定 Python 对象，使内部可变性线程安全。

调用 CPython C API 只有在操作系统线程显式附加到解释器运行时的情况下才是合法的。在支持 GIL 的构建中，这发生在获取 GIL 时。而在自由线程构建中没有 GIL，但相同的 C 宏（在支持 GIL 的构建中用于释放或获取 GIL）会改为请求解释器将线程附加到 Python 运行时，并且可以同时有多个线程附加。详见 PEP 703 获取更多关于线程如何以类似于在支持 GIL 的构建中释放/获取 GIL 的方式附加和分离到解释器运行时的背景信息。

在支持 GIL 的构建中，PyO3 使用 Python<'py> 类型和 'py 生命周期来表示持有全局解释器锁。在自由线程构建中，持有 'py 生命周期仅表示当前线程已附加到 Python 解释器——其他线程可以同时与解释器交互。

附加到运行时

你仍然需要获取一个 'py 生命周期来与 Python 对象交互或调用 CPython C API。如果你尚未附加到 Python 运行时，可以使用 Python::attach 函数注册线程。通过 Python threading 模块创建的线程不需要手动执行此操作，当 CPython 调用你的扩展时，pyo3 会自动设置 Python<'py> 令牌。

分离以避免挂起和死锁

自由线程构建在以下情况下会触发全局同步事件：

  • 在垃圾回收期间，以获得引用计数和对象间引用的全局一致视图
  • 在 Python 3.13 中，当第一个后台线程启动时，以标记某些对象为“永生”（immortal）
  • 当调用 sys.settrace 或 sys.setprofile 时，以对正在运行的代码对象和线程进行插桩
  • 在调用 os.fork() 期间，以确保整个进程状态的一致性。

此列表并非详尽无遗，未来 Python 版本中可能出现其他触发全局同步事件的场景。

因此，你应该在与支持 GIL 构建中相同的场景下使用 Python::detach 从解释器运行时分离：在执行不需要 CPython 运行时的长时间任务时，或在执行需要重新附加到运行时的任何任务时（参见指南章节了解此问题的详细说明）。在前一种情况下，你会观察到其他线程因等待长时间任务完成而挂起；在后一种情况下，当运行时触发全局同步事件后，某线程尝试重新附加时会遇到死锁，而生成该线程的主线程会阻止同步事件完成。

可变 pyclass 实例的多线程访问异常与恐慌

附加到 pyclass 实例的数据通过类似 RefCell 的运行时借用检查模式来防止并发访问。类似于 RefCell，PyO3 会在可变借用时引发异常（或某些情况下引发 panic）以强制执行独占访问。在 PyO3 中，过去通过 Python::detach 释放 GIL 或从 &mut self 调用接受 &self 的 Python 方法时就可能产生此类 panic（参见关于内部可变性的文档），但现在在自由线程 Python 中，由于没有 GIL 来锁定来自 Python 的对可变借用数据的并发访问，因此从 Python 代码中触发这些 panic 的机会更多了。触发此问题最直接的方法是使用 Python 的 threading 模块，在多个线程中同时调用一个可变借用 pyclass 的 Rust 函数。例如，考虑以下实现：

use pyo3::prelude::*;
#[pyclass]
#[derive(Default)]
struct ThreadIter {
    count: usize,
}

#[pymethods]
impl ThreadIter {
    #[new]
    pub fn new() -> Self {
        Default::default()
    }

    fn __next__(&mut self, py: Python<'_>) -> usize {
        self.count += 1;
        self.count
    }
}

然后，如果在 Python 中执行如下代码：

import concurrent.futures
from my_module import ThreadIter

i = ThreadIter()

def increment():
    next(i)

with concurrent.futures.ThreadPoolExecutor(max_workers=16) as tpe:
    futures = [tpe.submit(increment) for _ in range(100)]
    [f.result() for f in futures]

我们将看到一个异常：

Traceback (most recent call last)
  File "example.py", line 5, in <module>
    next(i)
RuntimeError: Already borrowed

未来版本的 PyO3 可能会允许用户为可变的 pyclass 定义选择不同的语义，从而可以选择启用某种锁机制来模拟 GIL（如果需要的话）。目前，你应该显式添加锁机制，可能使用条件编译或临界区 API，以避免与 GIL 产生死锁。

无法使用受限 API 构建扩展

自由线程构建使用了全新的 ABI，目前还没有针对自由线程 ABI 的受限 API 等价物。这意味着，如果你的 crate 依赖于使用 abi3 功能或某个 abi3-pyxx 功能的 PyO3，那么在使用自由线程解释器构建扩展时，PyO3 将打印警告并忽略该设置。

这意味着，如果你的包利用了受限 API 提供的 ABI 向前兼容性，从而只为每个版本上传一个 wheel 文件，那么你需要更新发布流程，额外上传一个特定版本的自由线程 wheel 文件。

有关支持多个不同 Python 版本（包括自由线程构建）的更多详细信息，请参见指南部分。

线程安全的单次初始化

为了确保数据仅初始化一次，请使用 PyOnceLock 类型，它类似于 std::sync::OnceLock，并且在有线程阻塞等待其他线程完成初始化时，会自动从 Python 解释器分离，从而帮助避免死锁。如果已经在使用 OnceLock，且替换为 PyOnceLock 不现实，则可以使用 OnceLockExt 扩展 trait，它提供了 OnceLockExt::get_or_init_py_attached 方法，在阻塞时自动从解释器分离，行为与 PyOnceLock 相同。以下是使用 PyOnceLock 单次初始化一个持有 Py<PyDict> 的运行时缓存的示例：

use pyo3::prelude::*;
use pyo3::sync::PyOnceLock;
use pyo3::types::PyDict;

let cache: PyOnceLock<Py<PyDict>> = PyOnceLock::new();

Python::attach(|py| {
    // 保证仅被调用一次
    cache.get_or_init(py, || PyDict::new(py).unbind())
});

在某些需要函数精确执行一次的场景中，可以将 OnceExt trait 引入作用域。OnceExt 为 std::sync::Once 添加了 OnceExt::call_once_py_attached 和 OnceExt::call_once_force_py_attached 方法，使其能够在已连接到 Python 解释器的上下文中使用 Once。这些函数类似于 Once::call_once、Once::call_once_force，但额外接受一个 Python<'py> token。所有这些函数在阻塞前会从解释器分离，并在执行函数之前重新连接，从而避免不使用 PyO3 扩展 trait 时可能出现的死锁。以下是使用 Once 而非 PyOnceLock 实现相同功能的示例：

use pyo3::prelude::*;
use std::sync::Once;
use pyo3::sync::OnceExt;
use pyo3::types::PyDict;

struct RuntimeCache {
    once: Once,
    cache: Option<Py<PyDict>>
}

let mut cache = RuntimeCache {
    once: Once::new(),
    cache: None
};

Python::attach(|py| {
    // 保证仅被调用一次
    cache.once.call_once_py_attached(py, || {
        cache.cache = Some(PyDict::new(py).unbind());
    });
});

GILProtected 未被导出

GILProtected 是 PyO3 中一个（已弃用）类型，它通过利用 GIL 来锁定其他线程的并发访问，从而实现对静态数据的可变访问。在自由线程 Python 中没有 GIL，因此你需要将此类型替换为其他形式的锁机制。在大多数情况下，使用 std::sync::atomic 中的类型或 std::sync::Mutex 就足够了。

之前：

#![allow(deprecated)]
fn main() {
#[cfg(not(Py_GIL_DISABLED))] {
use pyo3::prelude::*;
use pyo3::sync::GILProtected;
use pyo3::types::{PyDict, PyNone};
use std::cell::RefCell;

static OBJECTS: GILProtected<RefCell<Vec<Py<PyDict>>>> =
    GILProtected::new(RefCell::new(Vec::new()));

Python::attach(|py| {
    // 占位符，代表执行任意 Python 代码
    let d = PyDict::new(py);
    d.set_item(PyNone::get(py), PyNone::get(py)).unwrap();
    OBJECTS.get(py).borrow_mut().push(d.unbind());
});
}}

之后：

use pyo3::prelude::*;
fn main() {
use pyo3::types::{PyDict, PyNone};
use std::sync::Mutex;

static OBJECTS: Mutex<Vec<Py<PyDict>>> = Mutex::new(Vec::new());

Python::attach(|py| {
    // 占位符，代表执行任意 Python 代码
    let d = PyDict::new(py);
    d.set_item(PyNone::get(py), PyNone::get(py)).unwrap();
    // 与任何 `Mutex` 使用方式一样，应尽可能缩短加锁时间
    // 这里仅在向 `Vec` 添加元素时加锁
    OBJECTS.lock().unwrap().push(d.unbind());
});
}

如果你在持有锁的同时执行了任意 Python 代码，应导入 MutexExt trait 并使用 lock_py_attached 方法代替 lock。这能确保 Python 运行时启动的全局同步事件可以继续进行，从而避免与解释器可能发生死锁。
