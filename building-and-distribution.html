PyO3 user guide

构建与分发

本章将详细介绍如何使用 PyO3 构建和分发项目。具体实现方式取决于项目的类型：是以 Rust 实现的 Python 模块，还是嵌入 Python 的 Rust 可执行程序。对于这两类项目，都存在一些共性问题，例如目标 Python 版本的选择和应使用的链接器参数。

本章内容面向已经阅读过 PyO3 README 的用户，依次介绍针对 Python 模块和 Rust 可执行程序可做的各种选择，最后还有一节关于使用 PyO3 进行交叉编译的内容。

另有一个专门的子章节介绍支持多个 Python 版本的方法。

配置 Python 版本

PyO3 使用一个构建脚本（由 pyo3-build-config crate 支持）来确定 Python 版本并设置正确的链接器参数。默认情况下，它会按以下顺序尝试：

  • 当前激活的 Python 虚拟环境。
  • python 可执行文件（如果它是 Python 3 解释器）。
  • python3 可执行文件。

你可以通过设置 PYO3_PYTHON 环境变量来覆盖 Python 解释器，例如：PYO3_PYTHON=python3.7、PYO3_PYTHON=/usr/bin/python3.9，甚至是 PyPy 解释器 PYO3_PYTHON=pypy3。

一旦定位到 Python 解释器，pyo3-build-config 就会运行该解释器，查询 sysconfig 模块中的信息，以配置其余的编译过程。

要验证 PyO3 将使用的配置，可以设置环境变量 PYO3_PRINT_CONFIG=1 后运行编译。以下是一个示例输出：

$ PYO3_PRINT_CONFIG=1 cargo build
   Compiling pyo3 v0.14.1 (/home/david/dev/pyo3)
error: failed to run custom build command for `pyo3 v0.14.1 (/home/david/dev/pyo3)`

Caused by:
  process didn't exit successfully: `/home/david/dev/pyo3/target/debug/build/pyo3-7a8cf4fe22e959b7/build-script-build` (exit status: 101)
  --- stdout
  cargo:rerun-if-env-changed=PYO3_CROSS
  cargo:rerun-if-env-changed=PYO3_CROSS_LIB_DIR
  cargo:rerun-if-env-changed=PYO3_CROSS_PYTHON_VERSION
  cargo:rerun-if-env-changed=PYO3_PRINT_CONFIG

  -- PYO3_PRINT_CONFIG=1 is set, printing configuration and halting compile --
  implementation=CPython
  version=3.8
  shared=true
  abi3=false
  lib_name=python3.8
  lib_dir=/usr/lib
  executable=/usr/bin/python
  pointer_width=64
  build_flags=
  suppress_build_script_link_lines=false

PYO3_ENVIRONMENT_SIGNATURE 环境变量可在其值更改时触发重新构建，除此之外没有其他作用。

进阶：配置文件

如果将上述 PYO3_PRINT_CONFIG 输出的配置保存为文件，就可以手动修改内容，并通过 PYO3_CONFIG_FILE 环境变量重新导入 PyO3。

如果你的构建环境足够特殊，导致 PyO3 默认的配置检测机制失效，使用配置文件可以提供足够的灵活性以使 PyO3 正常工作。有关支持的完整选项，请参阅 InterpreterConfig 结构体 的文档。

构建 Python 扩展模块

Python 扩展模块的编译方式取决于其目标操作系统（和架构）。除了多种操作系统（和架构）之外，还有许多活跃支持的不同 Python 版本。上传至 PyPI 的包通常希望上传覆盖多种 OS/arch/version 组合的预编译“wheel”包，以便所有平台上的用户都不需要自行编译。包提供者可以选择启用“abi3”受限 Python API，这使得他们的 wheel 包可以在多个 Python 版本上使用，从而减少需要编译的 wheel 数量，但会限制可用的功能。

有多种方法可以实现这一点：可以使用 cargo 来构建扩展模块（并辅以一些手动操作，操作方式随操作系统不同而异）。PyO3 生态系统提供了两种打包工具 maturin 和 setuptools-rust，它们可以屏蔽操作系统差异，并支持构建用于 PyPI 上传的 wheel 包。

PyO3 提供了一些 Cargo 特性（feature）来配置构建 Python 扩展模块的项目：

  • extension-module 特性，构建 Python 扩展模块时必须启用。
  • abi3 特性及其版本特定的 abi3-pyXY 变体，用于启用受限 Python API，以便在单个 wheel 中支持多个 Python 版本。

本节将先分别介绍这两种打包工具，然后描述不使用这些工具时的手动构建方法。接着解释 extension-module 特性，最后介绍 PyO3 的 abi3 特性。

打包工具

PyO3 生态系统主要有两个工具可用来抽象 Python 扩展模块的开发流程：

  • maturin 是一个命令行工具，用于构建、打包和上传 Python 模块。它对项目结构有明确的约定，因此几乎无需额外配置。对于从零开始构建 Python 扩展且不需要灵活性的用户来说，这是一个理想的选择。
  • setuptools-rust 是 setuptools 的插件，为 setup.py 配置文件添加了额外的关键字参数。相比 maturin 需要更多配置，但这为那些需将 Rust 添加到现有 Python 包（不能满足 maturin 约束）的用户提供了更大的灵活性。

请查阅各项目的文档，了解如何开始使用以及如何上传 wheel 到 PyPI 的完整细节。需要注意的是，虽然 maturin 可以开箱即用地构建符合 manylinux 标准的 wheel，但 setuptools-rust 需要更多工作，依赖 Docker 来实现这一目标。

PyO3 仓库中还有 maturin-starter 和 setuptools-rust-starter 示例。

手动构建

要手动构建基于 PyO3 的 Python 扩展模块，首先在使用了 PyO3 的 extension-module 特性且具有 cdylib crate 类型的库项目中正常运行 cargo build。

构建完成后，将 Cargo target/ 目录中的共享库创建符号链接（或复制并重命名）到目标输出目录：

  • 在 macOS 上，将 libyour_module.dylib 重命名为 your_module.so。
  • 在 Windows 上，将 libyour_module.dll 重命名为 your_module.pyd。
  • 在 Linux 上，将 libyour_module.so 重命名为 your_module.so。

然后在输出目录中打开 Python shell，就可以执行 import your_module 了。

如果要将库打包分发，应在文件名中包含平台标签以标明该库所编译的目标 Python 解释器。这可以防止不兼容的解释器尝试导入你的库。如果你正在为 PyPy 编译，则必须包含平台标签，否则 PyPy 会忽略该模块。

Bazel 构建

要在 Bazel 中使用 PyO3，需要手动配置 PyO3、PyO3-ffi 和 PyO3-macros，特别是确保使用正确的 Python 标志来编译你打算使用的版本。例如参考：

  1. github.com/abrisco/rules_pyo3 —— 用于构建扩展模块的通用规则。
  2. github.com/OliverFM/pytorch_with_gazelle —— 一个使用 PyO3、PyTorch 和 Gazelle 生成 Python 构建文件的最小示例仓库。
  3. github.com/TheButlah/rules_pyo3 —— 稍显过时。

平台标签

除了上述建议的 .so 或 .pyd 扩展名外（依操作系统而定），你还可以在共享库扩展名前加上平台标签，以表明它兼容的解释器。你可以通过 sysconfig 模块查询解释器的平台标签。以下是一些示例输出：

# Python 3.10 上的 macOS
.cpython-310-darwin.so

# Linux 上的 PyPy 7.3 (对应 Python 3.9)
$ python -c 'import sysconfig; print(sysconfig.get_config_var("EXT_SUFFIX"))'
.pypy39-pp73-x86_64-linux-gnu.so

因此，例如在 macOS 上 CPython 3.10 的有效模块库名称为 your_module.cpython-310-darwin.so，而在 Linux 上为 PyPy 7.3 编译的等效名称为 your_module.pypy39-pp73-x86_64-linux-gnu.so。

有关平台标签的更多背景信息，请参阅 PEP 3149。

macOS

在 macOS 上，由于 extension-module 特性禁用了对 libpython 的链接（见下一节），需要设置一些额外的链接器参数。maturin 和 setuptools-rust 都会自动为 PyO3 传递这些参数，但手动构建的项目需要直接设置这些参数才能支持 macOS。

设置正确链接器参数最简单的方法是在项目中添加一个 build.rs，内容如下：

fn main() {
    pyo3_build_config::add_extension_module_link_args();
}

记得同时将 pyo3-build-config 添加到 Cargo.toml 的 build-dependencies 部分。

另一种不使用 pyo3-build-config 的方法是在 cargo 配置文件（如 .cargo/config.toml）中添加以下内容：

[target.x86_64-apple-darwin]
rustflags = [
  "-C", "link-arg=-undefined",
  "-C", "link-arg=dynamic_lookup",
]

[target.aarch64-apple-darwin]
rustflags = [
  "-C", "link-arg=-undefined",
  "-C", "link-arg=dynamic_lookup",
]
```使用 MacOS 系统自带的 python3（`/usr/bin/python3`，而非通过 homebrew、pyenv、nix 等方式安装的 Python）可能会导致运行时错误，例如 `Library not loaded: @rpath/Python3.framework/Versions/3.8/Python3`。

设置正确链接参数最简单的方法是添加一个 `build.rs` 文件，内容如下：

```rust,ignore
fn main() {
    pyo3_build_config::add_python_framework_link_args();
}

或者也可以在 .cargo/config.toml 中添加如下配置：

[build]
rustflags = [
  "-C", "link-args=-Wl,-rpath,/Library/Developer/CommandLineTools/Library/Frameworks",
]

关于 MacOS 链接问题的更多讨论和解决方案，请参见此问题。

最后，请注意在 MacOS 上使用 extension-module 特性时，如果不加上 --no-default-features 标志，cargo test 将会失败（详见 常见问题解答）。

extension-module 特性

PyO3 的 extension-module 特性用于在 Unix 目标平台上禁用对 libpython 的链接。

这是必要的，因为默认情况下 PyO3 会链接到 libpython，这使得二进制文件、测试和示例可以“开箱即用”。然而，为了符合 manylinux 标准，Unix 上的 Python 扩展模块不得链接 libpython。

不链接 libpython 的缺点是，那些通常需要嵌入 Python 的二进制文件、测试和示例将无法构建。如果你在一个项目中同时包含扩展模块和其他输出，则需要使用可选的 Cargo 特性，在构建非扩展模块时禁用 extension-module。有关示例解决方案，请参见 常见问题解答。

Py_LIMITED_API/abi3

默认情况下，Python 扩展模块只能用于与其编译时所使用的 Python 版本相同的环境中。例如，为 Python 3.5 构建的扩展模块无法在 Python 3.8 中导入。PEP 384 引入了“受限 Python API”的概念，它拥有稳定的 ABI，使用该 API 构建的扩展模块可以在多个 Python 版本中使用。这也被称为 abi3。

使用受限 Python API 构建扩展模块的优势在于，软件包分发者只需为每个操作系统/架构构建并分发一个版本，用户就可以在其所有符合最低版本要求的 Python 版本上安装使用。缺点是 PyO3 无法使用那些依赖于已知确切 Python 版本才能启用的优化。是否对此敏感取决于你的扩展模块的具体情况。你也可以设计你的扩展模块，使其既可分发 abi3 轮子，又允许从源码编译的用户获得额外的优化 —— 请参阅本指南中的 多 Python 版本支持 部分，特别是 #[cfg(Py_LIMITED_API)] 标志。

在构建 Python 轮子包时使用 abi3，需要完成以下三个步骤：

  1. 在 pyo3 中启用 abi3 特性。这确保 pyo3 只调用属于稳定 API 的 Python C-API 函数，并且在 Windows 上也确保项目链接到正确的共享对象（其他平台无需特殊行为）：

    [dependencies]
pyo3 = { {{#PYO3_CRATE_VERSION}}, features = ["abi3"] }

  2. 确保构建出的共享对象被正确标记为 abi3。这通过告知构建系统你正在使用受限 API 来实现。maturin >= 0.9.0 和 setuptools-rust >= 0.11.4 均支持 abi3 轮子。

    更多信息请参见 相关 PR 和 另一个 PR。

  3. 确保 .whl 文件被正确标记为 abi3。对于使用 setuptools 的项目，可通过向 setup.py bdist_wheel 传递 --py-limited-api=cp3x 参数实现（其中 x 是轮子所支持的最低 Python 版本，例如支持 Python 3.5 时使用 --py-limited-api=cp35）。

abi3 的最低 Python 版本

由于单个 abi3 轮子可用于多个不同的 Python 版本，PyO3 提供了 abi3-py37、abi3-py38、abi3-py39 等特性标志来设置你的 abi3 轮子所需的最低 Python 版本。例如，如果你设置了 abi3-py37 特性，你的扩展轮子就可以在 Python 3.7 及以上版本中使用。maturin 和 setuptools-rust 会将轮子命名为类似 my-extension-1.0-cp37-abi3-manylinux2020_x86_64.whl 的形式。

由于你的扩展模块可能在多个不同的 Python 版本中运行，有时你可能需要在运行时检查 Python 版本来定制行为。请参阅本指南 支持运行时多 Python 版本 的相关部分。

PyO3 仅能将你的扩展模块链接到等于或低于你当前主机 Python 版本的 abi3 版本。例如，如果你设置了 abi3-py38 并尝试使用 Python 3.7 主机编译，构建将失败。

注意：如果你同时设置了多个 abi3 版本特性标志，最低版本始终生效。例如，当同时设置 abi3-py37 和 abi3-py38 时，PyO3 将构建一个支持 Python 3.7 及以上版本的轮子。

在没有 Python 解释器的情况下构建 abi3 扩展

作为一个高级特性，你可以通过设置环境变量 PYO3_NO_PYTHON 在不调用 Python 解释器的情况下构建 PyO3 轮子。此外，如果构建主机上的 Python 解释器未找到、版本过旧或不可用，PyO3 仍会在显示警告信息后尝试编译 abi3 扩展模块。在类 Unix 系统上，此功能无条件可用；在 Windows 上，你还必须设置 RUSTFLAGS 环境变量，使其包含 -L native=/path/to/python/libs，以便链接器能找到 python3.lib。

如果 python3.dll 导入库不可用，可以启用实验性的 generate-import-lib crate 特性，PyO3 将自动创建并使用所需库。

注意：MSVC 目标需要 llvm-dlltool 工具（LLVM binutils 的一部分）位于 PATH 中，才能使自动导入库生成功能正常工作。

缺失的功能

由于 Python API 的限制，当为 abi3 编译时，部分 pyo3 功能无法使用，包括：

  • #[pyo3(text_signature = "...")] 在 Python 3.10 之前不支持类。
  • 类上的 dict 和 weakref 选项在 Python 3.9 之前不受支持。
  • 缓冲区 API 在 Python 3.11 之前不受支持。
  • 依赖于已知确切 Python 版本的优化功能。

在 Rust 中嵌入 Python

如果你想在 Rust 程序中嵌入 Python 解释器，有两种模式可选：动态嵌入和静态嵌入。我们将在接下来的部分介绍这两种模式，它们也会影响你的程序分发方式。与其自行学习如何实现，你可能更愿意使用像 PyOxidizer 这样的项目，将你的应用程序及其所有依赖打包成单个文件。

PyO3 会根据你配置 PyO3 使用的 Python 发行版（见上文）是否包含共享库或静态库，自动在这两种链接模式之间切换。静态库通常出现在从源码编译且未使用 --enable-shared 配置选项的 Python 发行版中。

动态嵌入 Python 解释器

动态嵌入 Python 解释器比静态嵌入容易得多。这通过将你的程序链接到 Python 共享库（如 UNIX 上的 libpython.3.9.so 或 Windows 上的 python39.dll）来实现。Python 解释器的实现位于共享库中。这意味着当操作系统运行你的 Rust 程序时，也需要能够找到 Python 共享库。

这种嵌入方式非常适合需要访问 Python 解释器的 Rust 测试。对于安装在 Python 虚拟环境（virtualenv）中的 Rust 软件也非常适用，因为虚拟环境会设置合适的环境变量以定位正确的 Python 共享库。

若要将程序分发给非技术用户，你还需要考虑将 Python 共享库一并包含在发布包中，并设置包装脚本以设置正确的环境变量（如 UNIX 上的 LD_LIBRARY_PATH 或 Windows 上的 PATH）。

请注意：PyPy 无法被嵌入到 Rust（或其他任何软件）中。对此的支持正在 PyPy 问题追踪器 中跟踪。

静态嵌入 Python 解释器

静态嵌入 Python 解释器意味着将 Python 静态库的内容直接包含在你的 Rust 二进制文件中。这意味着你在分发程序时只需要提供二进制文件：Python 解释器已经内嵌在二进制文件中！

在 Windows 上几乎从不做静态链接，因此 Python 发行版通常不包含静态库。以下信息仅适用于 UNIX 系统。

Python 静态库通常名为 libpython.a。静态链接存在许多复杂问题，如下所列。因此，PyO3 目前尚未对该嵌入模式提供一流支持。更多详情以及遇到问题时的讨论，请参见 PyO3 GitHub 上的 issue 416。

当静态嵌入解释器时，auto-initialize 功能会被刻意禁用，因为新用户在运行测试程序时常常会无意中执行静态链接。而使用动态嵌入方式尝试 PyO3 则要简单得多。

已知的复杂问题包括：

  • 为了导入已编译的扩展模块（例如其他用 Rust 或 C 编写的扩展模块），你的二进制文件在编译期间必须设置正确的链接器标志，以导出 libpython.a 的原始内容，供扩展模块使用（例如 -Wl,--export-dynamic）。

  • 创建 libpython.a 所使用的 C 编译器和编译标志必须与你的 Rust 编译器和标志兼容，否则会出现编译失败。

    明显不同的编译器版本可能会出现如下错误：

    lto1: fatal error: bytecode stream in file 'rust-numpy/target/release/deps/libpyo3-6a7fb2ed970dbf26.rlib' generated with LTO version 6.0 instead of the expected 6.2

    标志不匹配可能导致如下错误：

    /usr/bin/ld: /usr/lib/gcc/x86_64-linux-gnu/9/../../../x86_64-linux-gnu/libpython3.9.a(zlibmodule.o): relocation R_X86_64_32 against `.data' can not be used when making a PIE object; recompile with -fPIE

如果你在静态链接解释器时遇到上述或其他问题，请在 PyO3 GitHub 的 issue 416 中进行讨论。希望将来此讨论能积累足够的信息和解决方案，使得 PyO3 能够为静态嵌入提供一流支持。

在嵌入 Python 解释器时导入你的模块

当你运行带有嵌入解释器的 Rust 二进制文件时，任何通过 #[pymodule] 创建的模块都无法直接被导入，除非在初始化嵌入解释器之前将其添加到一个名为 PyImport_Inittab 的表中。否则，嵌入解释器中的 Python 语句（如 import your_new_module）将失败。你可以在初始化 Python 解释器前调用宏 append_to_inittab，将你的模块加入该表。（Python 解释器会在调用 Python::initialize、with_embedded_python_interpreter 或在启用 auto-initialize 功能时调用 Python::attach 时被初始化。）

交叉编译

得益于 Rust 强大的交叉编译支持，使用 PyO3 进行交叉编译相对简单。开始前，你需要准备以下软件：

  • 目标平台的工具链。
  • 在 Cargo .config 中为你的目标平台和所用工具链配置合适的选项。
  • 已为目标平台编译好的 Python 解释器（构建 “abi3” 扩展模块时可选）。
  • 主机平台上的 Python 解释器，并通过 PATH 或设置 PYO3_PYTHON 变量可访问（构建 “abi3” 扩展模块时可选）。

完成上述准备后，你可以使用 Cargo 的 --target 标志来构建交叉编译的 PyO3 模块。PyO3 的构建脚本会根据你的主机和目标平台自动检测是否正在进行交叉编译。

在交叉编译时，PyO3 的构建脚本无法运行目标平台的 Python 解释器来查询配置，因此你可能需要设置一些额外的环境变量：

  • PYO3_CROSS：存在时强制 PyO3 按交叉编译模式配置。
  • PYO3_CROSS_LIB_DIR：可设为包含目标平台 libpython 动态共享对象（DSO）及对应 _sysconfigdata*.py 文件（类 Unix 目标）或 Python DLL 导入库（Windows 目标）的目录。仅当输出二进制文件需要显式链接 libpython（例如目标为 Windows 和 Android 或嵌入 Python 解释器）或必须从 _sysconfigdata*.py 获取解释器配置时才需要设置该变量。
  • PYO3_CROSS_PYTHON_VERSION：目标 Python 安装的主版本和次版本（如 3.9）。仅当 PyO3 无法从 abi3-py3* 功能中确定目标版本，或未设置 PYO3_CROSS_LIB_DIR，或 PYO3_CROSS_LIB_DIR 中存在多个 Python 版本时，才需要设置该变量。
  • PYO3_CROSS_PYTHON_IMPLEMENTATION：目标 Python 安装的实现名称（“CPython” 或 “PyPy”）。默认假设为 CPython，除非未设置此变量且 PyO3 可通过 _sysconfigdata*.py 获取解释器配置（当为类 Unix 目标设置了 PYO3_CROSS_LIB_DIR 时）。

一个实验性的 pyo3 crate 功能 generate-import-lib 允许用户在不设置 PYO3_CROSS_LIB_DIR 环境变量或提供任何 Windows Python 库文件的情况下，交叉编译 Windows 平台的扩展模块。它通过外部的 python3-dll-a crate 为 MinGW-w64 和 MSVC 编译目标生成 Python DLL 的导入库。python3-dll-a 使用 binutils 的 dlltool 程序为 MinGW-w64 目标生成 DLL 导入库。可以通过设置 PYO3_MINGW_DLLTOOL 环境变量，覆盖交叉目标的默认 dlltool 命令名。注意：MSVC 目标要求主机系统上安装有 LLVM binutils 或 MSVC 构建工具。更具体地说，当目标为 *-pc-windows-msvc 时，python3-dll-a 要求 llvm-dlltool 或 lib.exe 可执行文件在 PATH 中。若设置了 ZIG_COMMAND 环境变量指向已安装的 Zig 程序（如 "zig" 或 "python -m ziglang"），则可使用 Zig 编译器替代 llvm-dlltool。

一个示例可能如下（假设目标 sysroot 位于 /home/pyo3/cross/sysroot，目标为 armv7）：

export PYO3_CROSS_LIB_DIR="/home/pyo3/cross/sysroot/usr/lib"

cargo build --target armv7-unknown-linux-gnueabihf

如果交叉库目录中存在多个 Python 版本，且无法设置更精确的位置来同时包含 libpython DSO 和 _sysconfigdata*.py 文件，可以显式指定所需版本：

export PYO3_CROSS_PYTHON_VERSION=3.8
export PYO3_CROSS_LIB_DIR="/home/pyo3/cross/sysroot/usr/lib"

cargo build --target armv7-unknown-linux-gnueabihf

另一个使用相同 sysroot 但构建 Windows 目标的例子：

export PYO3_CROSS_PYTHON_VERSION=3.9
export PYO3_CROSS_LIB_DIR="/home/pyo3/cross/sysroot/usr/lib"

cargo build --target x86_64-pc-windows-gnu

在上述示例中，也可以启用任意 abi3-py3* 功能来替代设置 PYO3_CROSS_PYTHON_VERSION。

在为 Unix 和 macOS 目标交叉编译扩展模块时，或在为 Windows 交叉编译扩展模块且启用了实验性 crate 功能 generate-import-lib 时，通常可以省略 PYO3_CROSS_LIB_DIR。
