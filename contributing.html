PyO3 user guide

贡献指南

感谢您有兴趣为 PyO3 做出贡献！我们欢迎所有人——请考虑阅读我们的行为准则，以帮助我们保持积极和包容的社区环境。

如果您正在寻找贡献的方向，请前往“开始贡献”部分。如果已经确定了具体的议题并需要了解开发流程，您可能会发现“编写 Pull Request”部分很有帮助。

如果您想熟悉代码库，请参阅 Architecture.md。

开始贡献

欢迎参与 PyO3 的任何感兴趣的方面。我们使用 GitHub issues 来记录所有 bug 和想法。如果您想处理某个 issue，可以请求将其分配给您。

您可以在此处浏览 PyO3 非公开部分的 API：here。

以下部分还提供了关于如何开始为 PyO3 做出贡献的具体建议。

搭建开发环境

要开发和使用 PyO3，您需要在系统上安装 Python 和 Rust。

  • 我们建议使用 rustup 来根据项目选择特定的工具链。
  • Pyenv 也非常推荐，用于选择特定的 Python 版本。
  • virtualenv 也可以与 Pyenv 一起使用（或单独使用），以使用特定安装的 Python 版本。
  • [nox][nox] 用于自动化许多 CI 任务。

使用 nox 进行测试、检查等

[Nox][nox] 用于自动化许多 CI 任务，可在本地编码时用于执行验证任务。我们建议通过 nox 运行这些操作，以便使用我们首选的配置选项。您可以使用 pip 将 nox 安装到全局 Python 环境中：pip install nox，或者（推荐方式）使用 [pipx][pipx]：pip install pipx，pipx install nox。

我们实现的主要 nox 命令包括：

  • nox -s test 将运行完整的 Rust 和 Python 推荐测试套件（>10 分钟）
  • nox -s test-rust -- skip-full 将运行简短的 Rust 测试套件（2-3 分钟）
  • nox -s ruff 将检查 Python 的代码风格并应用标准格式规则
  • nox -s rustfmt 将检查基本的 Rust 代码风格并应用标准格式规则
  • nox -s rumdl 将检查指南中的 Markdown
  • nox -s clippy 将运行 clippy 并对 Rust 风格提出建议
  • nox -s bench 将对您的 Rust 代码进行性能基准测试
  • nox -s codspeed 将运行我们的 Rust 和 Python 性能测试套件
  • nox -s coverage 将分析测试覆盖率并输出 coverage.json（或者使用 nox -s coverage lcov 输出 lcov.info）
  • nox -s check-guide 将使用 [lychee][lychee] 检查指南和文档注释中的所有链接。

使用 nox -l 可列出所有可运行的子命令。

UI 测试

PyO3 使用 [trybuild][trybuild] 来开发 UI 测试，用于捕获 Rust 编译器在某些宏功能上的错误消息。

由于这些 UI 测试有多种功能组合，当需要全部更新时（例如升级新的 Rust 编译器版本），可以使用 update-ui-tests nox 会话：

nox -s update-ui-tests

帮助用户识别 bug

PyO3 Discord 服务器 活跃着许多新用户，他们常常是刚接触 PyO3，甚至是完全新手的 Rust 开发者。帮助他们调试是熟悉 PyO3 代码库的好方法。

帮助他人通常会暴露出 bug、文档不足或缺失的 API。建议立即创建 GitHub issues 来记录这些问题，以便设计并实施解决方案！

实现已准备好开发的议题

对于解决方案明确且尚未有人开始工作的 issue，我们使用 needs-implementer 标签。

即使您不清楚解决方案也无需害怕！PyO3 的核心贡献者很乐意指导您，回答您的问题，帮助您编写解决方案。

协助编写优秀的文档

PyO3 拥有一本用户指南（使用 mdbook 编写）以及常规的 Rust API 文档。我们的目标是让这两者都详尽、易于理解且保持最新。欢迎提交 Pull Request 来修复拼写错误、调整措辞、添加示例等。

目前文档方面有一些特别需要帮助的重点领域：

  • 请求改进文档的 issue 使用 documentation 标签进行跟踪。
  • 并非所有 API 在创建时都有文档或示例。我们的目标是为所有 PyO3 API 提供文档（#306）。如果您发现某个 API 缺少文档，请编写一份并提交 PR！

要构建文档（包含所有特性），请先安装 [nox][nox]，然后运行：

nox -s docs -- open

文档测试 (Doctests)

我们在文档中使用了很多代码块。修改文档后，请运行 cargo test --doc 检查文档测试是否仍能通过，或者运行 cargo test 来运行包含文档测试在内的所有 Rust 测试。有关文档测试的指南，参见 https://doc.rust-lang.org/rustdoc/documentation-tests.html。

构建用户指南

您可以通过 mdbook 在本地构建用户指南进行预览。

首先，安装 [mdbook][mdbook]、[mdbook-tabs][mdbook-tabs] 插件以及 [nox][nox]。然后运行：

nox -s build-guide -- --open

要检查指南中所有链接是否有效，请额外安装 [lychee][lychee]，并改用 check-guide 会话：

nox -s check-guide

编写 Pull Request

在编写 PR 时，以下几点需要注意。

测试与持续集成

PyO3 仓库使用 GitHub Actions。如果 CI 不成功，PR 将无法合并。所有 Rust 和 Python 代码的格式化、检查和测试都会被验证（如果格式化失败，CI 管道会提前终止以节省资源）。此外，Rust 代码中不允许任何警告（使用 RUSTFLAGS="-D warnings"）。

测试会在所有支持的 Python 版本下使用最新的稳定版 Rust 编译器运行，同时也会针对 Python 3.9 和最低支持的 Rust 版本运行。

如果您添加了新功能，请将其加入 Cargo.toml 中的 full feature，以便 CI 能够对其进行测试。

您也可以使用 nox 自行运行 CI 管道组件，请参见上方的测试部分。

记录变更

我们使用 towncrier 为每次发布生成 CHANGELOG。

为了让您的更改被包含进发布日志，请在 newsfragments 目录中创建一个（或多个）新闻条目。有效的新闻条目应保存为 <PR>.<CATEGORY>.md，其中 <PR> 是 Pull Request 编号，<CATEGORY> 是以下之一：

  • packaging - 依赖项变更及 Python / Rust 版本兼容性变更
  • added - 新功能
  • changed - 已存在但已修改或已弃用的功能
  • removed - 已移除的功能
  • fixed - 属于 bug 修复的变更

仅文档更新的 PR 不需要新闻条目；请以 docs: 开头 PR 标题以跳过检查。

风格指南

泛型代码

PyO3 提供大量泛型 API 以提高可用性。但这可能会导致泛型代码膨胀。在合理的情况下，尽量为泛型函数实现一个具体的子部分。这有两种形式：

  • 如果该具体子部分不会被其他函数重复使用，则命名为 inner 并将其保留在函数内部作为局部函数。
  • 如果该具体子部分会被其他函数复用，则优先命名为 _foo，并在源码中直接放在 foo 下方（其中 foo 是原来的泛型函数）。

FFI 调用

PyO3 使用原始指针对 Python 的 C API 进行大量 FFI 调用。尽可能避免在表达式中使用指向临时对象的指针：

// 危险
pyo3::ffi::Something(name.to_object(py).as_ptr());

// 因为以下重构会导致悬垂指针（use-after-free）错误：
let name = name.to_object(py).as_ptr();
pyo3::ffi::Something(name)

相反，建议在传递给 FFI 函数前先绑定安全的、拥有所有权的 PyObject 包装器：

let name: PyObject = name.to_object(py);
pyo3::ffi::Something(name.as_ptr())
// 当 name 离开作用域时会自动释放

Python 与 Rust 版本支持策略

PyO3 旨在保持足够的兼容性，使得大多数常见包管理器能够方便地打包基于 PyO3 构建的 Python 扩展。

为简化包维护者的工作，PyO3 承诺尽可能仅在同时调整最低支持的 Rust 和 Python 版本时进行变更。此类升级仅会在 0.x 版本中进行，大约每年一次，在最旧支持的 Python 版本达到生命周期终点之后。（可查看 https://endoflife.date/python 获取明确的时间表。）

以下是每种语言 PR 所需兼容性的指导原则。

PythonPyO3 支持所有官方支持的 Python 版本，以及最新的 PyPy3 发行版。所有这些版本都会在 CI 中进行测试。

添加对新 CPython 版本的支持

如果你计划添加对 CPython 预发布版本的支持，以下是（非完整）检查清单：

  • 等待最后一个 alpha 版本（通常为 alpha7），因为在第一个 beta 版本之前，ABI 无法得到保证
  • 在 .github/workflows/ci.yml 中添加 prerelease_ver-dev（例如 3.14-dev），并在 noxfile.py、pyo3-ffi/Cargo.toml 的 [package.metadata.cpython] 中的 max-version、以及 pyo3-ffi/build.rs 的 max 字段中更新版本
  • 为该版本添加一个新的 abi3-prerelease 特性（例如 abi3-py314）
    • 在 pyo3-build-config/Cargo.toml 中，将 abi3-most_current_stable 设为 [“abi3-prerelease”]，abi3-prerelease 设为 [“abi3”]
    • 在 pyo3-ffi/Cargo.toml 中，将 abi3-most_current_stable 设为 [“abi3-prerelease”, “pyo3-build-config/abi3-most_current_stable”]，abi3-prerelease 设为 [“abi3”, “pyo3-build-config/abi3-prerelease”]
    • 在 Cargo.toml 中，将 abi3-most_current_stable 设为 [“abi3-prerelease”, “pyo3-ffi/abi3-most_current_stable”]，abi3-prerelease 设为 [“abi3”, “pyo3-ffi/abi3-prerelease”]
  • 使用 #[cfg(Py_prerelease])（例如 #[cfg(Py_3_14)]）和 #[cfg(not(Py_prerelease))] 来表示 CPython 稳定分支与预发布版本之间的变更
  • 不要为 CPython 头文件中以 _ 为前缀的任何函数、结构体或全局变量添加 Rust 绑定
  • 可联系 @ngoldbaum 和 @davidhewitt 获取帮助

Rust

PyO3 致力于使用最新的 Rust 语言特性，以保持其实现尽可能高效。

最低支持的 Rust 版本将在升级 Python 和 Rust 版本的发布时决定。届时，最低支持的 Rust 版本将不会高于当前 Debian、RHEL 和 Alpine Linux 发行版中提供的最低 Rust 版本。

CI 会同时测试最新的稳定 Rust 版本和最低支持的 Rust 版本。由于 Rust 的稳定性保证，这足以确认对中间所有 Rust 版本的支持。

基准测试

PyO3 有两组基准测试用于评估其性能的某些方面。当前基准测试套件非常小——如果你有兴趣帮助扩展它，请提交包含新基准的 PR！

首先，pyo3-benches 子目录中包含基于 Rust 的基准测试。你可以使用以下命令运行这些基准测试：

nox -s bench

其次，pytests 子目录中包含一个基于 Python 的基准测试。你可以点击此处了解更多信息。

代码覆盖率

你可以查看哪些代码被 PyO3 的测试覆盖，哪些没有。我们目标是实现 100% 覆盖率——如果你发现覆盖率不足，请检查并添加测试！

  • 首先，确保已安装 llvm-cov cargo 插件。在使用 nox 之前，你可能需要先通过 cargo 运行一次该插件。
cargo install cargo-llvm-cov
cargo llvm-cov
  • 然后，通过以下命令生成 lcov.info 文件
nox -s coverage -- lcov

你可以安装 IDE 插件来查看覆盖率。例如，如果你使用 VSCode：

  • 安装 coverage-gutters 插件。
  • 在 VSCode 的 settings.json 中添加以下设置：
{
    "coverage-gutters.coverageFileNames": [
        "lcov.info",
        "cov.xml",
        "coverage.xml",
    ],
    "coverage-gutters.showLineCoverage": true
}
  • 现在你应该能看到被测试代码的绿色高亮，以及未被测试代码的红色高亮。
