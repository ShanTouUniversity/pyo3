PyO3 user guide

类的自定义

Python 的对象模型定义了多种协议来控制不同的对象行为，例如序列、映射和数字协议。Python 类通过实现“魔法”方法（magic methods）来支持这些协议，例如 __str__ 或 __repr__。由于这些方法的名称两侧都有双下划线，它们也被称为“dunder”方法。

PyO3 允许在 #[pymethods] 中以与普通 Python 类相同的方式实现每个魔法方法，但有几点显著区别：

  • __new__ 和 __init__ 被 #[new] 属性 所替代。
  • __del__ 当前尚未支持，但未来可能会支持。
  • __buffer__ 和 __release_buffer__ 目前不支持，取而代之的是 PyO3 支持 __getbuffer__ 和 __releasebuffer__ 方法（这些方法早于 PEP 688），未来可能会发生变化。
  • PyO3 添加了 __traverse__ 和 __clear__ 方法用于控制垃圾回收。
  • 由于 PyO3 基于 Python C-API 实现，许多魔法方法需要具有特定的 C 函数签名，并被放入类类型对象的特殊“槽位”（slots）中。这限制了这些方法允许的参数和返回类型。这些限制在下文中有详细列出。

如果某个魔法方法不在上述列表中（例如 __init_subclass__），那么它在 PyO3 中应当可以直接使用。如果不是这样，请提交错误报告。

PyO3 处理的魔法方法

如果在 #[pymethods] 中定义的函数是一个需要特殊处理的已知魔法方法，该函数将被自动放置到 Python 类型对象的正确槽位中。函数名称遵循 #[pymethods] 的常规命名规则：如果存在 #[pyo3(name = "...")] 属性，则使用它；否则使用 Rust 函数名。

PyO3 处理的魔法方法与 此页面 上的标准 Python 方法非常相似，尤其是那些具有槽位定义的子集，如 此处定义。

当 PyO3 处理某个魔法方法时，与普通 #[pymethods] 相比有以下几个变化：

  • Rust 函数签名必须符合该魔法方法的要求。
  • 不允许使用 #[pyo3(signature = (...)] 和 #[pyo3(text_signature = "...")] 属性。

以下部分列出了 PyO3 实现特殊处理的所有魔法方法。所给的签名应按如下方式理解：

  • 所有方法都以接收器作为第一个参数，表示为 <self>。它可以是 &self、&mut self 或 Bound 引用（例如 self_: PyRef<'_, Self> 和 self_: PyRefMut<'_, Self>），如 此处所述。
  • 始终允许将可选的 Python<'py> 参数作为第一个参数。
  • 返回值可以选择性地用 PyResult 包裹。
  • object 意味着任何可以从中提取自 Python 对象（作为参数）或将转换为 Python 对象（作为返回值）的类型都可以使用。
  • 其他类型必须与给定的类型匹配，例如 __richcmp__ 的第二个参数应为 pyo3::basic::CompareOp。
  • 对于比较和算术方法，提取错误不会以异常形式传播，而是返回 NotImplemented。
  • 对于某些魔法方法，返回值不受 PyO3 限制，但由 Python 解释器检查。例如，__str__ 必须返回字符串对象。这表示为 object (Python type)。

基本对象自定义

  • __str__(<self>) -> object (str)

  • __repr__(<self>) -> object (str)

  • __hash__(<self>) -> isize

    比较相等的对象必须具有相同的哈希值。可以返回任何最多 64 位的类型来代替 isize，PyO3 会自动将其转换为 isize（对无符号类型如 u64 和 usize 进行包装）。

    禁用 Python 默认哈希

    默认情况下，所有 #[pyclass] 类型都有来自 Python 的默认哈希实现。不应可哈希的类型可以通过将 __hash__ 设置为 None 来覆盖此行为。这与纯 Python 类的机制相同。实现方式如下：

    use pyo3::prelude::*;

#[pyclass]
struct NotHashable {}

#[pymethods]
impl NotHashable {
    #[classattr]
    const __hash__: Option<Py<PyAny>> = None;
}

  • __lt__(<self>, object) -> object

  • __le__(<self>, object) -> object

  • __eq__(<self>, object) -> object

  • __ne__(<self>, object) -> object

  • __gt__(<self>, object) -> object

  • __ge__(<self>, object) -> object

    分别实现 Python 的“富比较”运算符 <、<=、==、!=、> 和 >=。

    注意：实现其中任一方法将导致 Python 不再生成默认的 XXXXXXXXXX 实现，因此在实现时应考虑同时实现 XXXXXXXXXX。

    返回类型 返回类型通常为 `bool` 或 `PyResult`，但也可以返回任何 Python 对象。

  • __richcmp__(<self>, object, pyo3::basic::CompareOp) -> object

    在单个方法中实现 Python 比较操作（==、!=、<、<=、> 和 >=）。CompareOp 参数表示当前执行的比较操作。你可以使用 CompareOp::matches 将 Rust 的 std::cmp::Ordering 结果适配为请求的比较操作。

    此方法不能与 XXXXXXXX、XXXXXXXX、XXXXXXXX、XXXXXXXX、XXXXXXXX 或 XXXXXXXX 组合实现。

    注意：实现 XXXXXXXXXXXXX 将导致 Python 不再生成默认的 XXXXXXXXXX 实现，因此在实现 XXXXXXXXXXXXX 时应考虑同时实现 XXXXXXXXXX。

    返回类型 返回类型通常为 `PyResult`，但可以返回任何 Python 对象。

    如果你希望某些操作未实现，可以在某些操作中返回 py.NotImplemented()：

    use pyo3::class::basic::CompareOp;
use pyo3::types::PyNotImplemented;

use pyo3::prelude::*;
use pyo3::BoundObject;

#[pyclass]
struct Number(i32);

#[pymethods]
impl Number {
    fn __richcmp__<'py>(&self, other: &Self, op: CompareOp, py: Python<'py>) -> PyResult<Borrowed<'py, 'py, PyAny>> {
        match op {
            CompareOp::Eq => Ok((self.0 == other.0).into_pyobject(py)?.into_any()),
            CompareOp::Ne => Ok((self.0 != other.0).into_pyobject(py)?.into_any()),
            _ => Ok(PyNotImplemented::get(py).into_any()),
        }
    }
}

    如果第二个参数 object 不符合签名中指定的类型，生成的代码将自动 return NotImplemented。

  • __getattr__(<self>, object) -> object

  • __getattribute__(<self>, object) -> object

    `__getattr__` 和 `__getattribute__` 的区别 与 Python 中一样，只有在通过正常属性查找未找到属性时才会调用 `__getattr__`。而 `__getattribute__` 则会在 *每次* 属性访问时被调用。如果它需要访问 `self` 上的现有属性，必须非常小心以避免无限递归，并使用 `baseclass.__getattribute__()`。

  • __setattr__(<self>, value: object) -> ()

  • __delattr__(<self>, object) -> ()

    覆盖属性访问。

  • __bool__(<self>) -> bool

    决定对象的“真值性”。

  • __call__(<self>, ...) -> object —— 这里可以像普通 pymethods 一样定义任意参数列表。

可迭代对象

可以通过以下方法定义迭代器：

  • __iter__(<self>) -> object
  • __next__(<self>) -> Option<object> 或 IterNextOutput （参见详情）

从 __next__ 返回 None 表示没有更多项了。

示例：

use pyo3::prelude::*;
use std::sync::Mutex;

#[pyclass]
struct MyIterator {
    iter: Mutex<Box<dyn Iterator<Item = Py<PyAny>> + Send>>,
}

#[pymethods]
impl MyIterator {
    fn __iter__(slf: PyRef<'_, Self>) -> PyRef<'_, Self> {
        slf
    }

    fn __next__(slf: PyRefMut<'_, Self>) -> Option<Py<PyAny>> {
        slf.iter.lock().unwrap().next()
    }
}

在许多情况下，你将区分被迭代的类型（即 可迭代对象）和它提供的迭代器。在这种情况下，可迭代对象只需实现 __iter__()，而迭代器必须同时实现 __iter__() 和 __next__()。例如：

use pyo3::prelude::*;

#[pyclass]
struct Iter {
    inner: std::vec::IntoIter<usize>,
}

#[pymethods]
impl Iter {
    fn __iter__(slf: PyRef<'_, Self>) -> PyRef<'_, Self> {
        slf
    }

    fn __next__(mut slf: PyRefMut<'_, Self>) -> Option<usize> {
        slf.inner.next()
    }
}

#[pyclass]
struct Container {
    iter: Vec<usize>,
}

#[pymethods]
impl Container {
    fn __iter__(slf: PyRef<'_, Self>) -> PyResult<Py<Iter>> {
        let iter = Iter {
            inner: slf.iter.clone().into_iter(),
        };
        Py::new(slf.py(), iter)
    }
}

Python::attach(|py| {
    let container = Container { iter: vec![1, 2, 3, 4] };
    let inst = pyo3::Py::new(py, container).unwrap();
    pyo3::py_run!(py, inst, "assert list(inst) == [1, 2, 3, 4]");
    pyo3::py_run!(py, inst, "assert list(iter(iter(inst))) == [1, 2, 3, 4]");
});

有关 Python 迭代协议的更多细节，请查阅 库文档中的“迭代器类型”部分。

从迭代中返回值本指南到目前为止展示了如何使用 Option<T> 在迭代过程中产生值。在 Python 中，生成器也可以返回一个值，这是通过抛出 StopIteration 异常实现的。要在 Rust 中表达这一点，应返回一个 PyResult::Err，并将 PyStopIteration 作为错误值。

可等待对象

  • __await__(<self>) -> object
  • __aiter__(<self>) -> object
  • __anext__(<self>) -> Option<object>

映射与序列类型

本节中的魔术方法可用于实现 Python 容器类型。Python 中主要有两类容器：“映射”（如 dict），其键是任意的；以及“序列”（如 list 和 tuple），其键为整数。

PyO3 所基于的 Python C-API 为序列和映射分别设置了独立的“插槽”。但在纯 Python 中编写 class 时，实现上并无此类区分——例如，一个 __getitem__ 的实现会同时填充映射和序列的插槽。

默认情况下，PyO3 会复现 Python 的行为，即同时填充映射和序列的插槽。这对于与 Python 行为一致的“简单”情况以及序列类型是合理的，因为在序列中，映射插槽本来也会用于实现切片索引。

而映射类型通常不希望填充序列插槽。如果填充了这些插槽，可能会导致非预期的结果，例如：

  • 该映射类型将能成功转换为 PySequence，这可能导致使用者错误地处理该类型。
  • Python 为序列提供了默认的 __iter__ 实现，它会从 0 开始连续调用 __getitem__ 传入正整数索引，直到抛出 IndexError 为止。除非映射的键仅为连续的正整数，否则此 __iter__ 实现很可能不是预期行为。

使用 #[pyclass(mapping)] 注解可指示 PyO3 仅填充映射插槽，而让序列插槽为空。这将作用于 __getitem__、__setitem__ 和 __delitem__。

使用 #[pyclass(sequence)] 注解可指示 PyO3 在实现 __len__ 时填充 sq_length 插槽而非 mp_length 插槽。这有助于诸如 numpy 等库将此类识别为序列，但也会导致 CPython 在将负索引传递给 __getitem__ 前自动加上序列长度。（对于切片操作，__getitem__、__setitem__ 和 __delitem__ 仍然使用映射插槽。）

  • __len__(<self>) -> usize

    实现内建函数 len()。

  • __contains__(<self>, object) -> bool

    实现成员测试操作符。若 item 在 self 中应返回 true，否则返回 false。对于未定义 __contains__() 的对象，成员测试将简单地遍历序列直到找到匹配项。

    禁用 Python 默认的 contains 实现

    默认情况下，所有带有 __iter__ 方法的 #[pyclass] 类型都支持 in 操作符的默认实现。若某些类型不需要此行为，可通过将 __contains__ 设置为 None 来覆盖。这与纯 Python 类的机制相同。实现方式如下：

    use pyo3::prelude::*;

#[pyclass]
struct NoContains {}


#[pymethods]
impl NoContains {
    #[classattr]
    const __contains__: Option<Py<PyAny>> = None;
}

  • __getitem__(<self>, object) -> object

    实现 self[a] 元素的获取。

    注意：PyO3 不会特殊处理负整数索引。但对于带有 #[pyclass(sequence)] 的类，当通过 PySequence::get_item 访问负索引时，底层 C API 已经会将索引调整为正值。

  • __setitem__(<self>, object, object) -> ()

    实现对 self[a] 元素的赋值。仅当元素可被替换时才应实现此方法。

    对负索引的处理行为与 __getitem__ 相同。

  • __delitem__(<self>, object) -> ()

    实现删除 self[a] 元素。仅当元素可被删除时才应实现此方法。

    对负索引的处理行为与 __getitem__ 相同。

  • fn __concat__(&self, other: impl FromPyObject) -> PyResult<impl ToPyObject>

    连接两个序列。在尝试通过 __add__ 和 __radd__ 方法进行数值加法后，由 + 操作符使用。

  • fn __repeat__(&self, count: isize) -> PyResult<impl ToPyObject>

    将序列重复 count 次。在尝试通过 __mul__ 和 __rmul__ 方法进行数值乘法后，由 * 操作符使用。

  • fn __inplace_concat__(&self, other: impl FromPyObject) -> PyResult<impl ToPyObject>

    连接两个序列。在尝试通过 __iadd__ 方法进行数值加法后，由 += 操作符使用。

  • fn __inplace_repeat__(&self, count: isize) -> PyResult<impl ToPyObject>

    连接两个序列。在尝试通过 __imul__ 方法进行数值乘法后，由 *= 操作符使用。

垃圾回收器集成

如果你的类型持有对其他 Python 对象的引用，则需要与 Python 的垃圾回收器集成，以便 GC 能感知这些引用。为此，需实现 __traverse__ 和 __clear__ 两个方法。它们对应于 Python C API 中的 tp_traverse 和 tp_clear 插槽。__traverse__ 必须对每个指向其他 Python 对象的引用调用 visit.call()。__clear__ 必须清除对其他可变 Python 对象的所有引用（从而打破引用循环）。不可变引用无需清除，因为每个循环中必须至少包含一个可变引用。

  • __traverse__(<self>, pyo3::class::gc::PyVisit<'_>) -> Result<(), pyo3::class::gc::PyTraverseError>
  • __clear__(<self>) -> ()

注意：__traverse__ 不能与 #[pyo3(warn(...))] 一起使用。

示例：

use pyo3::prelude::*;
use pyo3::PyTraverseError;
use pyo3::gc::PyVisit;


#[pyclass]
struct ClassWithGCSupport {
    obj: Option<Py<PyAny>>,
}


#[pymethods]
impl ClassWithGCSupport {
    fn __traverse__(&self, visit: PyVisit<'_>) -> Result<(), PyTraverseError> {
        visit.call(&self.obj)?;
        Ok(())
    }


    fn __clear__(&mut self) {
        // 清除引用，这会减少引用计数。
        self.obj = None;
    }
}

通常，__traverse__ 的实现除了调用 visit.call 外不应做其他任何事。最关键的是，在 __traverse__ 的实现中禁止安全访问解释器，即 Python::attach 将引发 panic。> 注意：这些方法属于 C API，PyPy 不一定支持它们。如果你正在为 PyPy 构建程序，应测量内存占用情况，以确保没有失控的内存增长。请参阅 PyPy bug 跟踪器中的此问题。
