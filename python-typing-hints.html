PyO3 user guide

为您的 Python 包提供类型提示和 IDE 提示

PyO3 提供了一个易于使用的接口，用于在 Rust 中编写原生 Python 库。配套的 Maturin 允许您将它们构建并发布为一个包。然而，为了更好的用户体验，Python 库应为其所有公共实体提供类型提示和文档，以便 IDE 可以在开发期间显示这些信息，且类型分析工具（如 mypy）能够使用它们来正确验证代码。

目前解决此问题的最佳方案是手动维护 *.pyi 文件，并随包一起发布。

PyO3 正在致力于实现自动生成这些文件。有关自动生成当前状态的说明，请参阅 类型存根生成 文档。

pyi 文件简介

pyi 文件（“Python Interface”的缩写）在大多数相关文档中被称为“存根文件（stub files）”。关于它是什么的一个非常好的定义可以在 旧版 MyPy 文档 中找到：

存根文件只包含模块公共接口的描述，而不包含任何实现。

此外，官方 Python typing 文档中也有关于类型存根的 详尽文档。

大多数 Python 开发者可能已经遇到过它们，例如在尝试对任何内置类型使用 IDE 的“转到定义”功能时。例如，一些标准异常的定义如下所示：

class BaseException(object):
    args: Tuple[Any, ...]
    __cause__: BaseException | None
    __context__: BaseException | None
    __suppress_context__: bool
    __traceback__: TracebackType | None
    def __init__(self, *args: object) -> None: ...
    def __str__(self) -> str: ...
    def __repr__(self) -> str: ...
    def with_traceback(self: _TBE, tb: TracebackType | None) -> _TBE: ...


class SystemExit(BaseException):
    code: int


class Exception(BaseException): ...


class StopIteration(Exception):
    value: Any

如我们所见，这些不是包含实现的完整定义，而仅仅是接口的描述。通常这正是库用户所需的一切。

PEP 中怎么说？

在编写本文档时，pyi 文件在四个 PEP 中被提及。

PEP8 - Python 代码风格指南 - # 函数注解（最后一段）建议所有第三方库的创建者提供存根文件，作为类型检查工具了解包的信息来源。

（……）预计第三方库包的用户可能希望在其包上运行类型检查器。为此，PEP 484 建议使用存根文件：.pyi 文件，类型检查器会优先读取这些文件而不是相应的 .py 文件。（……）

PEP484 - 类型提示 - # 存根文件 将存根文件定义如下：

存根文件是仅用于类型检查器而非运行时的、包含类型提示的文件。

其中包含对它们的规范说明（强烈推荐阅读，因为它包含至少一个在普通 Python 代码中不会使用的特性），以及一些关于存根文件应如何存储的通用信息。

PEP561 - 分发和打包类型信息 详细描述了如何构建支持类型检查的包。特别是其中包含了存根文件必须如何分发才能让类型检查器使用的具体信息。

PEP560 - typing 模块与泛型类型的核心支持 描述了 Python 类型系统内部如何支持泛型的细节，包括运行时行为和与静态类型检查器的集成。

如何操作？

PEP561 承认了三种分发类型信息的方式：

  • inline —— 类型提示直接放在源码（py）文件中；
  • 独立的存根文件包 —— 类型提示放在 pyi 文件中，并发布在独立的单独包中；
  • 包内嵌存根文件 —— 类型提示放在 pyi 文件中，并与源文件一起发布在同一个包中。

第一种方式在 PyO3 中较为困难，因为我们没有 py 文件。待该方式被研究并实现必要的更改后，本文档将进行更新。

第二种方式容易实现，整个工作可以完全与主库代码分离。存根文件包的示例仓库可在 PEP561 参考部分 找到：存根包仓库

第三种方式将在下文描述。

将 pyi 文件包含在您的 PyO3/Maturin 构建包中

当源文件与存根文件位于同一包中时，它们应彼此相邻放置。我们需要一种方式通过 Maturin 实现这一点。此外，为了标记我们的包支持类型提示，我们需要在包中添加一个名为 py.typed 的空文件。

如果您不需要添加其他 Python 文件

如果除了 pyi 文件外您不需要向包中添加任何其他 Python 文件，Maturin 可以为您完成大部分工作。如 Maturin 指南 文档所述，您只需在项目根目录创建一个名为 <module_name>.pyi 的存根文件，其余工作将由 Maturin 自动完成。

my-rust-project/
├── Cargo.toml
├── my_project.pyi  # <<< 在此处为 my_project 模块中的 Rust 函数添加类型存根
├── pyproject.toml
└── src
    └── lib.rs

有关 pyi 文件的示例，请参见 my_project.pyi 内容 章节。

如果您需要其他 Python 文件

如果您需要向包中添加除 pyi 之外的其他 Python 文件，也可以做到，但需要额外的工作。Maturin 提供了一种方便的方式来向包中添加文件（文档）。您只需在 Cargo.toml 文件旁边创建一个与模块同名的文件夹（如需自定义，请参阅上述文档链接）。

文件结构如下：

my-project
├── Cargo.toml
├── my_project
│   ├── __init__.py
│   ├── my_project.pyi
│   ├── other_python_file.py
│   └── py.typed
├── pyproject.toml
├── Readme.md
└── src
    └── lib.rs

让我们更详细地说明包文件夹内的各个文件。

__init__.py 内容

现在我们自行指定包内容，因此必须提供 __init__.py 文件，以便该文件夹被识别为包，并且我们可以从中导入内容。我们可以始终使用 Maturin 在未指定 Python 源文件夹时为我们生成的内容。对于 PyO3 绑定，它应该是：

from .my_project import *

这样，我们原生模块公开的所有内容都可以直接从包中导入。

py.typed 文件要求

如 PEP561 所述：

希望支持其代码类型检查的包维护者，必须在其支持类型的包中添加一个名为 py.typed 的标记文件。该标记具有递归性：如果顶级包包含它，其所有子包也必须支持类型检查。

如果我们不包含该文件，某些 IDE 仍可能会使用我们的 pyi 文件显示提示，但类型检查器可能不会。MyPy 在这种情况下会报错：

error: Skipping analyzing "my_project": found module but no type hints or library stubs

该文件只是一个标记文件，因此应为空。

支持泛型

类型注解在 Python 中也可以是泛型的。它们在处理不同类型的同时保持类型安全性方面非常有用。通常，泛型类会继承自 typing.Generic 元类。

例如，以下 .pyi 文件定义了一个可以接受多种类型轮子的 Car：

from typing import Generic, TypeVar


W = TypeVar('W')


class Car(Generic[W]):
    def __init__(self, wheels: list[W]) -> None: ...


    def get_wheels(self) -> list[W]: ...


    def change_wheel(self, wheel_number: int, wheel: W) -> None: ...

这样一来，最终用户便可以使用变量如 truck: Car[SteelWheel] = ... 和 f1_car: Car[AlloyWheel] = ... 来指定类型。

此外，在 Python 3.12+ 中有一种特殊的语法用于指定泛型类型：

class Car[W]:
    def __init__(self, wheels: list[W]) -> None: ...


    def get_wheels(self) -> list[W]: ...
```#### 运行时行为

存根文件（`pyi`）仅对静态类型检查器有用，在运行时会被忽略。因此，即使在存根文件中指定了，PyO3 类也不会继承自 `typing.Generic`。

这可能会导致一些运行时问题，因为像 `f1_car: Car[AlloyWheel] = ...` 这样的变量注解会使 Python 调用未定义的魔术方法。

为克服此限制，实现者可以在 Rust 中将 `generic` 参数传递给 `pyclass`：

```rust ignore
#[pyclass(generic)]

高级用户

#[pyclass(generic)] 实现了一种非常简单的运行时行为，可接受任意泛型参数。高级用户可以选择手动为泛型类实现 __class_getitem__，以获得更精细的控制。

impl MyClass {
    #[classmethod]
    #[pyo3(signature = (key, /))]
    pub fn __class_getitem__(
        cls: &Bound<'_, PyType>,
        key: &Bound<'_, PyAny>,
    ) -> PyResult<Py<PyAny>> {
        /* 实现细节 */
    }
}

请注意，在实现 __class_getitem__ 时，使用 pyo3::types::PyGenericAlias 可能会有所帮助，因为它可以从 Rust 创建 types.GenericAlias 对象。
