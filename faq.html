PyO3 user guide

常见问题与故障排查

很抱歉您在使用 PyO3 时遇到问题。如果您无法在下方列表中找到问题的解决方案，也可以通过 GitHub Discussions 或 Discord 寻求帮助。

在使用 std::sync::OnceLock、std::sync::LazyLock、lazy_static 和 once_cell 时出现死锁

OnceLock、LazyLock 及其第三方前身实现机制依赖阻塞来确保仅有一个线程对它们进行初始化。由于 Python 解释器会引入额外的锁（Python GIL 和 GC 都可能要求所有其他线程暂停），这可能导致以下方式的死锁：

  1. 一个连接到 Python 解释器的线程（线程 A）开始初始化 OnceLock 的值。
  2. 初始化代码调用了某些 Python API，例如 Python::import，这会临时脱离解释器。
  3. 另一个线程（线程 B）连接到 Python 解释器，并尝试访问同一个 OnceLock 值。
  4. 线程 B 被阻塞，因为它在等待 OnceLock 的初始化锁释放。
  5. 在非自由线程模式的 Python 中，线程 A 现在也被阻塞，因为它需要重新连接到解释器（获取线程 B 仍然持有的 GIL）。
  6. 发生死锁。

PyO3 提供了一个名为 PyOnceLock 的结构体，它实现了基于这些类型的单次初始化 API，并避免了死锁。您还可以使用 OnceExt 和 OnceLockExt 扩展 trait，它们为此目的提供了标准库类型的新方法，从而避免与 Python 解释器死锁的风险。这意味着当您遇到上述描述的死锁问题时，可以使用它们来替代其他选择。有关 PyOnceLock 和 OnceExt 的更多详细信息和使用示例，请参阅相关文档。

无法运行 cargo test；或无法在 Cargo 工作区中构建：出现“Symbol not found”或“Undefined reference to _PyExc_SystemError”等链接错误

目前 #340 会导致在启用 extension-module 功能时 cargo test 因链接错误而失败。当在 Cargo 工作区中构建且另一个 crate 也使用 PyO3 时，也可能出现链接错误（见 #2521）。目前，我们有三种方法可以绕过这些问题。

  1. 将 extension-module 功能设为可选，使用 maturin develop --features "extension-module" 构建：

    [dependencies.pyo3]
{{#PYO3_CRATE_VERSION}}

[features]
extension-module = ["pyo3/extension-module"]

  2. 将 extension-module 功能设为可选并设为默认，使用 cargo test --no-default-features 运行测试：

    [dependencies.pyo3]
{{#PYO3_CRATE_VERSION}}

[features]
extension-module = ["pyo3/extension-module"]
default = ["extension-module"]

  3. 如果您正在使用 pyproject.toml 文件控制 maturin 设置，请添加以下部分：

    [tool.maturin]
features = ["pyo3/extension-module"]
# 或者对于 maturin 0.12:
# cargo-extra-args = ["--features", "pyo3/extension-module"]

无法运行 cargo test：无法在 tests/ 目录中的测试里找到您的 crate

Rust 书籍建议将集成测试放在 tests/ 目录中。

对于一个 extension-module 项目，如果您的 Cargo.toml 中将 crate-type 设置为了 "cdylib"，编译器将无法找到您的 crate，并显示如 E0432 或 E0463 的错误：

error[E0432]: unresolved import `my_crate`
 --> tests/test_my_crate.rs:1:5
  |
1 | use my_crate;
  |     ^^^^^^^^^^^^ no external crate `my_crate`

最佳的解决方案是让您的 crate 类型同时包含 rlib 和 cdylib：

# Cargo.toml
[lib]
crate-type = ["cdylib", "rlib"]

执行 Rust 代码时 Ctrl-C 无效

这是因为 Ctrl-C 会触发 SIGINT 信号，该信号由调用的 Python 进程处理，但 Python 仅设置一个标志，稍后才会处理。在从 Python 调用的 Rust 代码执行期间，该标志不会被检查，只有当控制权返回 Python 解释器时才会检查。

您可以通过调用 Python::check_signals 给 Python 解释器机会正确处理该信号。如果您有一个长时间运行的 Rust 函数，建议定期调用此函数，以便用户能够取消执行。

#[pyo3(get)] 克隆了我的字段

您可能有一个类似下面的嵌套结构体：

use pyo3::prelude::*;
#[pyclass]
#[derive(Clone)]
struct Inner {/* 字段省略 */}

#[pyclass]
struct Outer {
    #[pyo3(get)]
    inner: Inner,
}

#[pymethods]
impl Outer {
    #[new]
    fn __new__() -> Self {
        Self { inner: Inner {} }
    }
}

当 Python 代码访问 Outer 的字段时，PyO3 每次访问都会返回一个新对象（注意它们的地址不同）：

outer = Outer()

a = outer.inner
b = outer.inner

assert a is b, f"a: {a}\nb: {b}"

AssertionError: a: <builtins.Inner object at 0x00000238FFB9C7B0>
b: <builtins.Inner object at 0x00000238FFB9C830>

如果该字段是可变的，这可能会特别令人困惑，因为获取字段并修改它不会持久化——下一次访问会得到原始对象的全新克隆。遗憾的是，Python 和 Rust 在所有权上并不一致——如果 PyO3 向 Python 代码提供对（可能临时的）Rust 对象的引用，Python 代码就可能无限期地保留该引用。因此返回 Rust 对象需要克隆。

如果您不希望发生克隆，一种解决方法是将字段分配在 Python 堆上并存储对该对象的引用，使用 Py<...>：

use pyo3::prelude::*;
#[pyclass]
struct Inner {/* 字段省略 */}

#[pyclass]
struct Outer {
    inner: Py<Inner>,
}

#[pymethods]
impl Outer {
    #[new]
    fn __new__(py: Python<'_>) -> PyResult<Self> {
        Ok(Self {
            inner: Py::new(py, Inner {})?,
        })
    }

    #[getter]
    fn inner(&self, py: Python<'_>) -> Py<Inner> {
        self.inner.clone_ref(py)
    }
}

这次 a 和 b 确实是同一个对象：

outer = Outer()

a = outer.inner
b = outer.inner

assert a is b, f"a: {a}\nb: {b}"
print(f"a: {a}\nb: {b}")

a: <builtins.Inner object at 0x0000020044FCC670>
b: <builtins.Inner object at 0x0000020044FCC670>

这种方法的缺点是，任何对 Outer 结构体进行操作的 Rust 代码都可能需要连接到 Python 解释器才能处理 inner 字段。（如果 Inner 带有 #[pyclass(frozen)] 且实现了 Sync，则可以使用 Py::get 从 Py<Inner> 访问 Inner 内容而无需连接解释器。）

我想使用从依赖项重新导出的 pyo3 crate，但过程宏失败

所有 PyO3 过程宏（#[pyclass]、#[pyfunction]、#[derive(FromPyObject)] 等）都期望在 crate 根下能以 pyo3 这个名称访问到 pyo3 crate，这在 pyo3 是您 crate 的直接依赖时是正常的。

然而，当依赖项被重命名，或您的 crate 仅间接依赖于 pyo3 时，您需要让宏代码知道该在哪里查找这个 crate。这是通过 crate 属性完成的：

use pyo3::prelude::*;
pub extern crate pyo3;
mod reexported { pub use ::pyo3; }
#[allow(dead_code)]
#[pyclass]
#[pyo3(crate = "reexported::pyo3")]
struct MyClass;

我试图从 Rust 调用 Python，但得到了 STATUS_DLL_NOT_FOUND 或 STATUS_ENTRYPOINT_NOT_FOUND

这在 Windows 上发生，原因是链接 Python DLL 失败或链接了错误的 DLL。Windows 上的 Python DLL 通常命名为：

  • python3X.dll 对于 Python 3.X，例如 Python 3.10 对应 python310.dll
  • 使用 PyO3 的 abi3 功能时为 python3.dll

必须能够根据 Windows DLL 搜索顺序 找到该 DLL。实现此目的的一些方法包括：

  • 将 Python DLL 放在构建产物的同一文件夹中
  • 将包含 Python DLL 的目录添加到 PATH 环境变量中，例如 C:\Users\<You>\AppData\Local\Programs\Python\Python310
  • 如果此问题出现在您分发程序时，可以考虑使用 PyOxidizer 将其与二进制文件打包在一起

如果链接了错误的 DLL，可能是因为另一个程序将自己及其 Python DLL 添加到了 PATH。请调整 PATH 变量顺序，使正确的 DLL 优先。

注意：对 PATH（或任何其他环境变量）的更改在现有 shell 中不可见。请重启 shell 以使更改生效。

对于高级故障排查，可以使用 Dependency Walker 来诊断链接错误。
