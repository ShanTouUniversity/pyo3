PyO3 user guide

错误处理

本章简要介绍了 Rust 中的错误处理机制，以及 PyO3 如何将其与 Python 异常集成。

这涵盖足够的细节来创建一个 #[pyfunction]，该函数会从 Rust 中的错误引发 Python 异常。

指南后面有关于 Python 异常 的一节，更详细地介绍了异常类型。

表示 Python 异常

Rust 代码使用泛型 Result<T, E> 枚举来传播错误。错误类型 E 由代码作者选择，用于描述可能发生的具体错误。

PyO3 提供了 PyErr 类型，表示一个 Python 异常。如果某个 PyO3 API 可能导致 Python 异常被抛出，那么该 API 的返回类型将是 PyResult<T>，这是 Result<T, PyErr> 类型的别名。

总结如下：

  • 当 Python 异常被引发并由 PyO3 捕获时，该异常将存储在 PyResult 的 Err 变体中。
  • 通过 Rust 代码传递 Python 异常时，使用所有“常规”技术，例如 ? 操作符，其中错误类型为 PyErr。
  • 最后，当一个 PyResult 通过 PyO3 从 Rust 返回到 Python 时，如果结果是 Err 变体，则其中包含的异常将被引发。

（关于 Rust 错误处理和 ? 操作符有许多优秀的教程，因此本指南不会深入讲解仅限 Rust 的主题。）

从函数中抛出异常

如前所述，当包含 Err 的 PyResult 从 Rust 转移到 Python 时，PyO3 将引发其中包含的异常。

因此，要从 #[pyfunction] 中抛出异常，请将返回类型 T 更改为 PyResult<T>。当函数返回 Err 时，就会引发一个 Python 异常。（只要错误类型 E 实现了 From 转换为 PyErr，其他 Result<T, E> 类型也可以使用，请参见下文的 自定义 Rust 错误类型。）

这对 #[pymethods] 中的函数也有效。

例如，以下 check_positive 函数在输入为负数时引发 ValueError：

use pyo3::exceptions::PyValueError;
use pyo3::prelude::*;

#[pyfunction]
fn check_positive(x: i32) -> PyResult<()> {
    if x < 0 {
        Err(PyValueError::new_err("x is negative"))
    } else {
        Ok(())
    }
}

fn main(){
	Python::attach(|py|{
		let fun = pyo3::wrap_pyfunction!(check_positive, py).unwrap();
		fun.call1((-1,)).unwrap_err();
		fun.call1((1,)).unwrap();
	});
}

所有内置的 Python 异常类型都定义在 pyo3::exceptions 模块中。它们都有 new_err 构造函数，可以直接构建 PyErr，如上例所示。

自定义 Rust 错误类型

只要存在 std::from::From<E> for PyErr 的实现，PyO3 就会自动将 #[pyfunction] 返回的 Result<T, E> 转换为 PyResult<T>。Rust 标准库中的许多错误类型都以这种方式定义了 From 转换。

如果你处理的类型 E 是定义在第三方 crate 中的，请参阅下文的 外部 Rust 错误类型 一节，了解如何处理此类错误。

以下示例利用了 From<ParseIntError> for PyErr 的实现，在解析字符串为整数时引发异常：

use pyo3::prelude::*;
use std::num::ParseIntError;

#[pyfunction]
fn parse_int(x: &str) -> Result<usize, ParseIntError> {
    x.parse()
}

fn main() {
    Python::attach(|py| {
        let fun = pyo3::wrap_pyfunction!(parse_int, py).unwrap();
        let value: usize = fun.call1(("5",)).unwrap().extract().unwrap();
        assert_eq!(value, 5);
    });
}

当传入不包含浮点数的字符串时，引发的异常如下所示：

>>> parse_int("bar")
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
ValueError: invalid digit found in string

更完整的例子：下面的代码片段定义了一个名为 CustomIOError 的 Rust 错误类型。然后定义了 From<CustomIOError> for PyErr，返回一个表示 Python OSError 的 PyErr。因此，可以直接在 #[pyfunction] 的结果中使用此错误，并依赖转换机制将其传播为 Python 异常。

use pyo3::exceptions::PyOSError;
use pyo3::prelude::*;
use std::fmt;

#[derive(Debug)]
struct CustomIOError;

impl std::error::Error for CustomIOError {}

impl fmt::Display for CustomIOError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "Oh no!")
    }
}

impl std::convert::From<CustomIOError> for PyErr {
    fn from(err: CustomIOError) -> PyErr {
        PyOSError::new_err(err.to_string())
    }
}

pub struct Connection {/* ... */}

fn bind(addr: String) -> Result<Connection, CustomIOError> {
    if &addr == "0.0.0.0" {
        Err(CustomIOError)
    } else {
        Ok(Connection{ /* ... */})
    }
}

#[pyfunction]
fn connect(s: String) -> Result<(), CustomIOError> {
    bind(s)?;
    // etc.
    Ok(())
}

fn main() {
    Python::attach(|py| {
        let fun = pyo3::wrap_pyfunction!(connect, py).unwrap();
        let err = fun.call1(("0.0.0.0",)).unwrap_err();
        assert!(err.is_instance_of::<PyOSError>(py));
    });
}

如果希望延迟构建 Python 异常实例，可以改为实现 PyErrArguments trait。在这种情况下，实际的异常参数创建将推迟到需要 PyErr 时才进行。

最后需要注意的是，任何实现了 From 转换的错误 E 都可以与 ?（“try”）操作符一起使用。以下是上述 parse_int 函数的另一个实现版本，直接返回 PyResult：

use pyo3::prelude::*;

fn parse_int(s: String) -> PyResult<usize> {
    let x = s.parse()?;
    Ok(x)
}

use pyo3::exceptions::PyValueError;

fn main() {
    Python::attach(|py| {
        assert_eq!(parse_int(String::from("1")).unwrap(), 1);
        assert_eq!(parse_int(String::from("1337")).unwrap(), 1337);

        assert!(parse_int(String::from("-1"))
            .unwrap_err()
            .is_instance_of::<PyValueError>(py));
        assert!(parse_int(String::from("foo"))
            .unwrap_err()
            .is_instance_of::<PyValueError>(py));
        assert!(parse_int(String::from("13.37"))
            .unwrap_err()
            .is_instance_of::<PyValueError>(py));
    })
}

外部 Rust 错误类型

Rust 编译器不允许为在当前 crate 之外定义的类型实现 trait（这被称为“孤儿规则”）。

假设有一个在第三方代码中定义的类型 OtherError，要将其与 PyO3 集成，主要有两种策略：

  • 创建一个新的类型包装器，例如 MyOtherError。然后实现 From<MyOtherError> for PyErr（或 PyErrArguments），以及 From<OtherError> for MyOtherError。
  • 使用 Rust 的 Result 组合子，如 map_err，自由编写代码将 OtherError 转换为所需的类型。这种方式每次使用都需要样板代码，但提供了无限灵活性。

进一步说明新类型策略的关键技巧是从 #[pyfunction] 返回 Result<T, MyOtherError>。这意味着 PyO3 将利用 From<MyOtherError> for PyErr 来创建 Python 异常，而 #[pyfunction] 的实现可以通过 ? 自动将 OtherError 转换为 MyOtherError。

以下示例演示了某个假想第三方 crate some_crate 的情况，其中函数 get_x 返回 Result<i32, OtherError>：

mod some_crate {
  pub struct OtherError(());
  impl OtherError {
      pub fn message(&self) -> &'static str { "some error occurred" }
  }
  pub fn get_x() -> Result<i32, OtherError> { Ok(5) }
}

use pyo3::prelude::*;
use pyo3::exceptions::PyValueError;
use some_crate::{OtherError, get_x};

struct MyOtherError(OtherError);

impl From<MyOtherError> for PyErr {
    fn from(error: MyOtherError) -> Self {
        PyValueError::new_err(error.0.message())
    }
}

impl From<OtherError> for MyOtherError {
    fn from(other: OtherError) -> Self {
        Self(other)
    }
}

#[pyfunction]
fn wrapped_get_x() -> Result<i32, MyOtherError> {
    // get_x 是一个返回 Result<i32, OtherError> 的函数
    let x: i32 = get_x()?;
    Ok(x)
}

fn main() {
    Python::attach(|py| {
        let fun = pyo3::wrap_pyfunction!(wrapped_get_x, py).unwrap();
        let value: usize = fun.call0().unwrap().extract().unwrap();
        assert_eq!(value, 5);
    });
}

注意事项

在 Python 3.11 及更高版本中，可以向 Python 异常添加注释（notes），以便在打印异常时提供额外的调试信息。在 PyO3 中，你可以使用 PyErr 上的 add_note 方法实现此功能。
